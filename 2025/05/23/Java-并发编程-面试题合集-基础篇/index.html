🧵 线程基础篇

⭐⭐⭐ 超高频问题

1. Java 线程与操作系统线程是什么关系？

标准答案：

  • 一对一模型：主流 JVM（如 HotSpot）采用一对一线程模型，一个 Java 线程对应一个操作系统内核线程
  • 映射机制：JVM 通过 JNI 调用操作系统 API 创建和管理内核线程
  • 调度方式：线程调度由操作系统负责，JVM 不直接参与调度决策

深入理解：

实际应用：

  • 线程数量受操作系统限制（通常几千个）
  • 创建线程开销较大，适合使用线程池
  • 线程切换涉及内核态转换，有一定性能开销

2. 详细说明 Java 线程的 6 种状态及转换条件？

标准答案：

状态 说明 进入条件 退出条件
NEW 线程已创建但未启动 new Thread() 调用 start()
RUNNABLE 可运行状态 调用 start()后 阻塞/等待/结束
BLOCKED 等待监视器锁 进入 synchronized 块失败 获得锁
WAITING 无限期等待 wait()、join()、park() notify()、unpark()、中断
TIMED_WAITING 有时限等待 sleep()、wait(timeout) 超时、notify()、中断
TERMINATED 线程结束 run()执行完毕或异常 无（终态）

常见陷阱：

  • RUNNABLE 状态包含了 OS 的 Ready 和 Running 状态
  • 从 WAITING/TIMED_WAITING 唤醒后，需要重新竞争锁才能继续执行

3. wait()、notify()、notifyAll()的使用规则和原理？

标准答案：

使用规则：

  1. 必须在 synchronized 块中调用
  2. 必须由锁对象调用（synchronized(obj) -> obj.wait()）
  3. wait()会释放锁，notify()不会立即释放锁

原理机制：

面试重点：

  • wait()与 sleep()的区别：wait()释放锁，sleep()不释放锁
  • notify()与 notifyAll()的选择：优先使用 notifyAll()避免信号丢失
  • 虚假唤醒：必须在 while 循环中调用 wait()

🧠 Java 内存模型篇

⭐⭐⭐ 超高频问题

6. 什么是 Java 内存模型（JMM）？它解决了什么问题？

标准答案：

定义：JMM 是 Java 虚拟机规范的一部分，定义了线程和主内存之间的抽象关系，规定了多线程程序中变量的访问规则。

核心概念：

  • 主内存：所有线程共享的内存区域，存储对象实例、静态变量等
  • 工作内存：每个线程私有的内存区域，存储主内存变量的副本
  • 内存交互：通过 8 种原子操作完成主内存与工作内存的数据交换

解决的问题：

  1. 硬件差异：屏蔽不同 CPU 架构的内存模型差异
  2. 缓存一致性：解决多级缓存导致的数据不一致
  3. 指令重排序：规范编译器和处理器的优化边界
  4. 可见性保证：定义何时一个线程的修改对另一个线程可见

7. JMM 如何保证可见性、原子性、有序性？

详细解答：

可见性保证：

  • volatile 关键字：强制从主内存读取，修改后立即写回主内存
  • synchronized 关键字：获取锁时清空工作内存，释放锁时刷新到主内存
  • final 关键字：构造函数完成后，final 字段对所有线程可见

原子性保证：

  • 基本类型读写：long 和 double 的 64 位操作在某些平台可能非原子
  • synchronized 关键字：保证代码块的原子执行
  • Lock 接口：提供更灵活的原子性控制
  • Atomic 类：通过 CAS 操作保证原子性

有序性保证：

  • 程序次序规则：单线程内按代码顺序执行
  • volatile 关键字：禁止特定的指令重排序
  • synchronized 关键字：同步代码块串行执行
  • happens-before 规则：定义操作间的偏序关系

8. 详细解释 happens-before 规则？

核心规则：

  1. 程序次序规则：单线程内，前面的操作 happens-before 后面的操作
  2. 监视器锁定规则：unlock 操作 happens-before 同一个锁的 lock 操作
  3. volatile 变量规则：volatile 写 happens-before 后续的 volatile 读
  4. 线程启动规则：Thread.start()happens-before 该线程的所有操作
  5. 线程终止规则：线程的所有操作 happens-before join()的返回
  6. 中断规则：interrupt()happens-before 检测到中断事件
  7. 对象终结规则：构造函数完成 happens-before finalize()开始
  8. 传递性：A happens-before B，B happens-before C，则 A happens-before C

内存屏障类型：

  • LoadLoad 屏障：保证屏障前的 load 完成后，才执行屏障后的 load
  • StoreStore 屏障：保证屏障前的 store 完成后，才执行屏障后的 store

10. JMM 的 8 种内存交互操作是什么？

8 种原子操作：

  1. lock：作用于主内存变量，标记为线程独占
  2. unlock：作用于主内存变量，释放锁定状态
  3. read：从主内存读取变量到工作内存
  4. load：将 read 的值放入工作内存变量副本
  5. use：将工作内存变量值传递给执行引擎
  6. assign：将执行引擎的值赋给工作内存变量
  7. store：将工作内存变量值传送到主内存
  8. write：将 store 的值写入主内存变量

操作规则：

  • read 和 load、store 和 write 必须成对出现
  • 不允许线程丢弃最近的 assign 操作
  • 变量只能在主内存中”诞生”
  • 同一时刻只允许一个线程 lock 一个变量

14. synchronized 的锁优化有哪些？

锁升级机制：

详细说明：

1. 偏向锁：

  • 优化场景：只有一个线程反复获取同一个锁
  • 实现原理：在对象头记录偏向线程 ID，后续获取锁只需检查 ID
  • 类比：会议室的”专属预约牌”

2. 轻量级锁：

  • 优化场景：多线程交替获取锁，竞争不激烈
  • 实现原理：通过 CAS 操作尝试获取锁，失败则自旋等待
  • 类比：门口的”快速登记本”

3. 重量级锁：

  • 使用场景：竞争激烈，自旋效果不佳
  • 实现原理：依赖操作系统互斥量，线程阻塞等待
  • 类比：专门的管理员协调使用

其他优化：

  • 锁消除：JIT 编译器发现不可能共享的对象，消除锁
  • 锁粗化：扩大锁的范围，减少获取/释放锁的开销
  • 自适应自旋：根据历史情况动态调整自旋次数

常见误区

❌ 错误认知：

  • “volatile 能保证原子性”
  • “synchronized 很慢，要避免使用”
  • “wait()和 sleep()效果一样”

✅ 正确理解：

  • volatile 只保证可见性和有序性
  • JDK6 后 synchronized 已大幅优化
  • wait()会释放锁，sleep()不会
